# TinaIDE 开发指南

> 更新日期：2026-04-22

本指南只保留当前代码仍然成立的开发事实，避免继续引用已经过时的流程。

## 项目当前形态

TinaIDE 目前采用 `app + core/* + feature/* + tools/* + external/*` 的模块化结构：

- `app`：应用壳、首页导航、项目编辑器入口、启动流程
- `core/*`：编译、LSP、sysroot、工具链、Git、插件、存储、网络等复用能力
- `feature/*`：编辑器、终端、设置、AI、文件预览、帮助、工作区等用户功能
- `tools/*`：模板 APK、模板公共代码等辅助模块
- `external/*`：本地源码依赖与子模块

后端、管理端和部署运维代码已迁入私有仓库，不再随开源 Android 项目分发。
公开仓库只保留 `server/README.md` 作为边界说明。

默认编译与运行依赖的是：

- `Android sysroot`
- `native tina-toolchain`

可选的 Linux / PRoot 环境不再是基础编译必需项。

## 入口与模块认知

开发时先分清两个主入口：

- `MainPortalActivity`：项目列表、设置、帮助、插件、用户入口
- `MainActivity`：打开项目后的主编辑器工作区

如果你在调试“首页行为”，优先看 `MainPortalActivity`；如果你在调试编辑器 / 文件树 / 构建 / 底部面板，优先看 `MainActivity` 及其 Compose 状态层。

## 开发环境

### 必需

1. Android Studio 最新稳定版
2. JDK 17+
3. Git

### 可选

1. PowerShell 7+
2. Docker Desktop

只有在你需要重建 toolchain / sysroot / rootfs 资产时，Docker 才是必需的。

## 本地启动

### 推荐方式

```powershell
pwsh ./tools/build-apk.ps1 -Install
```

### 常用变体

```powershell
# 模拟器
pwsh ./tools/build-apk.ps1 -Abi x86 -Install

# Release
pwsh ./tools/build-apk.ps1 -Variant release
```

### 直接使用 Gradle

```bash
./gradlew :app:assembleArm64Debug
./gradlew :app:installArm64Debug
./gradlew "-Ptina.devAbi=x86_64" :app:assembleX86_64Debug
./gradlew "-Ptina.devAbi=x86_64" :app:installX86_64Debug
./gradlew :app:assembleDebugAllAbi
./gradlew :app:assembleReleaseAllAbi
```

本地默认只启用一个 ABI 变体：

- 默认 `arm64`
- 通过 `-Ptina.devAbi=x86_64` 切到模拟器 ABI
- 通过 `assemble*AllAbi` 或 `-Ptina.allAbi=true` 恢复双 ABI

## 首次启动与依赖安装

当前默认首次启动流程会进入依赖安装页，并安装：

1. `Android sysroot`
2. `native tina-toolchain`

只有显式进入 Linux 环境相关入口时，才会额外安装：

1. `Linux distro rootfs`
2. `Linux distro runtime metadata`
3. `PRoot guest toolchain`
4. 然后校验 / 安装 `Android sysroot`
5. 然后校验 / 安装 `native tina-toolchain`

对应事实源：

- `StartupFlowManager.kt`
- `DependencyInstallViewModel.kt`
- `AndroidNativeToolchainManager.kt`
- `AndroidSysrootManager.kt`
- `PRootBootstrap.kt`

## 编辑器 / 语言服务开发注意事项

当前编辑器语言服务并不是“所有文件都走 clangd”：

- C / C++：`LspEditorManager` + `LspClientSession`
- CMake：`CMakeLanguageServiceSession`
- Make / `.mk`：`MakeLanguageServiceSession`
- 其他语言：`LspPluginManager` 按插件 manifest 匹配

因此，遇到补全 / 悬停 / 跳转问题时，先确认自己调试的是哪一条路径，不要直接把问题归到 `clangd` 或 `core:lsp`。

## AI 工具开发落点

`feature:ai` 当前通过 `ToolInitializer` 批量向 `ToolRegistry` 注册工具，核心约定在：

- `feature/ai/.../AiTool.kt`
- `feature/ai/.../ToolRegistry.kt`
- `feature/ai/.../ToolInitializer.kt`
- `feature/ai/.../ToolParameterParser.kt`
- `feature/ai/.../ToolI18n.kt`

当前内置工具按类别组织为：

- `EDITOR`
- `FILE_SYSTEM`
- `CODE_ANALYSIS`
- `DIAGNOSTICS`
- `BUILD`
- `EXECUTION`
- `REFACTOR`
- `GIT`
- `TERMINAL`
- `WEB`
- `CUSTOM`

原先零散归类成“项目管理”的能力，当前分别落在 `FILE_SYSTEM`、
`CODE_ANALYSIS`、`GIT`、`TERMINAL` 等类别里。新增工具时，优先复用这些
入口，不要在 UI 层私自拼装“半工具化”逻辑。

## 帮助文档与仓库文档

App 内帮助中心不是直接读取 `docs/` 目录，而是通过：

- `feature/help/.../HelpRepository.kt`
- `feature/help/src/main/assets/help/*.md`

如果你改的是面向 App 内用户的帮助文档，需要同步检查 help assets；如果你改的是仓库协作文档，则更新 `docs/`。

## 运行资产维护

当前 native 工具链资产不是“手工丢几个压缩包”这么简单，而是受 `current.properties` 约束。

常用入口：

```powershell
# 同步 toolchain / sysroot 资产
pwsh ./tools/sync-tina-toolchain-assets.ps1 -Abi arm64 -Clean
pwsh ./tools/sync-tina-toolchain-assets.ps1 -Abi x86_64 -Clean
```

相关校验：

- `app/build.gradle.kts` 中的 `verifyTinaToolchainAssets`
- `app/src/<abi>/assets/tina-toolchain/current.properties`
- `app/src/<abi>/assets/android-sysroot/*.tar.xz`

## 开发时优先遵守的约束

### 1. 用户可见文案必须走国际化

不要把用户能看到的中文 / 英文直接写进 Kotlin / Java 代码。

优先使用：

- `AppStrings.kt`
- `ResExt.kt`
- `TextResourceAliases.kt`
- `AppResourceAliases.kt`

### 2. 日志统一走项目日志封装

不要在业务代码中直接使用 `android.util.Log`。

当前构建脚本已提供守卫任务：

- `checkNoAndroidUtilLog`

统一使用 `Timber` / `TinaTimber`。

### 3. 模块边界优先清晰

- `feature/*` 尽量只表达用户功能，不互相直接耦合
- `core/*` 提供可复用能力
- `app` 负责装配，而不是继续堆积业务逻辑

### 4. 文档与代码一起改

涉及以下内容时，需要同步更新文档：

- 运行时安装流程
- 模块结构
- 构建命令
- 新增对外能力或删除旧能力

## Release 构建副作用

当前 Release 相关任务会触发额外行为：

- 自动递增 `version.properties`
- 备份 mapping 文件
- mapping 文件仅由公开构建逻辑做本地归档

因此，Release 构建仍不适合作为“纯只读验证命令”随手执行。若维护者需要外部
symbol 存储集成，应放在公开仓库之外维护。

## GitHub Actions 配置

### PR 自动构建

公开仓库可以直接使用 `pull_request` Workflow 自动构建 Debug APK，**不需要任何密钥**。

- Fork 提交 PR 也可以触发构建
- 产物会作为 artifact 上传
- 这类 Workflow 不要使用 release signing secrets

### Release / Tag 构建

如果你希望公开仓库在 tag 或手动 release 时产出正式 APK，请在 GitHub 仓库的
`Settings -> Secrets and variables -> Actions` 里配置：

- `TINAIDE_RELEASE_KEYSTORE_B64`
- `TINAIDE_RELEASE_KEYSTORE_PASSWORD`
- `TINAIDE_RELEASE_KEY_ALIAS`
- `TINAIDE_RELEASE_KEY_PASSWORD`

说明：

- `TINAIDE_RELEASE_KEYSTORE_B64` 是 `release.jks` 的 Base64 内容
- 其它几项分别对应签名口令和别名
- 这些值只用于 release/tag 构建，不要放进公开代码

### 已废弃：私有仓库派发

旧的“私有仓库构建 -> `repository_dispatch` 派发到公开仓库”流程已经废弃。
当前公开仓库只保留自身的 tag / `workflow_dispatch` 发布入口，不再需要
`TINAIDE_REPO_TOKEN`、`TINAIDE_REPO` 或 `PRO_REPO_TOKEN`。

## 代码提交流程

推荐的最小流程：

1. 修改代码
2. 运行受影响模块的测试或构建任务
3. 更新相关文档
4. 自查用户可见文案、日志、边界和回归点
5. 提交

Commit message 建议继续使用 Conventional Commits，例如：

```bash
git commit -m "docs(readme): align runtime docs with native toolchain flow"
git commit -m "fix(compile): handle missing toolchain metadata"
git commit -m "refactor(editor): split session persistence logic"
```

## 子模块注意事项

仓库包含多个 Git submodule。涉及子模块指针变更时，顺序必须是：

1. 先在子模块仓库提交并推送
2. 再回到主仓库提交子模块指针

否则主仓库 CI 在执行 `git submodule update --recursive` 时可能出现 `not our ref <sha>`。
