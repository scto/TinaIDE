# 代码补全

MobileIDE 当前的代码补全能力以 **clangd + LSP** 为核心，重点服务于 C / C++ / CMake 这类工程化场景。

## 什么时候会触发补全

在语言服务正常工作时，补全通常会在这些场景自动出现：

- 输入标识符前缀
- 输入成员访问（如 `obj.`、`ptr->`）
- 输入作用域解析（如 `std::`）
- 输入函数参数时联动显示签名帮助
- 输入 `#include` 相关内容时提供头文件候选

## 当前补全体验的重点

基于源码，当前补全相关能力不只是一份“候选列表”，还包括：

- **补全候选数量限制**
- **大小写敏感开关**
- **clangd 补全样式**
- **函数参数占位符模式**
- **签名帮助**
- **Inlay Hints（行内提示）**
- **Semantic Tokens（语义高亮）**

这些都可以在 **设置 → 语言服务器** 中看到并调整。

## 想让补全更准，最关键是什么

### 1. 项目先成功构建一次

这是最重要的一步。很多补全、跳转和诊断效果，都会在首次成功构建后明显更稳定。

### 2. 保证 `compile_commands.json` 可用

对于 CMake 项目，这份编译数据库非常关键。它决定了 clangd 是否知道：

- 头文件搜索路径
- 宏定义
- 编译参数
- 当前源文件所属目标

### 3. 包含路径与构建配置要真实有效

如果 `CMakeLists.txt` 或 Makefile 本身配置不对，补全也不可能长期准确。

## 与代码片段的关系

当前补全列表中除了 clangd 提供的语义候选，还可能出现：

- 应用内置片段
- 通过插件系统安装的代码片段

所以“补全”在 MobileIDE 里实际上是 **语义候选 + 片段候选** 的组合体验。

## 本地 LSP 与远程 LSP

补全可以运行在两种模式下：

- **本地 LSP** - 在设备本地完成分析
- **远程 LSP** - 在电脑侧完成分析

如果项目很大、手机性能有限，远程 LSP 往往能显著改善补全延迟和稳定性。

## 推荐配置思路

### 小项目

- 直接使用本地 LSP
- 保持默认候选数即可

### 中大型项目

- 优先保证 `compile_commands.json` 正确
- 视情况降低补全候选数量
- 必要时切换到远程 LSP

### 偏向“更强提示”的用户

- 开启签名帮助
- 开启 Inlay Hints
- 使用更详细的 clangd 补全样式

## 常见问题

### Q: 为什么完全没有补全？

优先检查：

1. 当前项目是否已经构建过
2. `compile_commands.json` 是否存在且是最新的
3. **设置 → 语言服务器** 中相关开关是否正常
4. 当前是否误切到了不可用的远程 LSP 配置

### Q: 为什么补全很多，但不准确？

通常说明 clangd 能工作，但：

- 编译数据库不完整
- 头文件路径配置不对
- 工程实际构建参数和编辑器看到的不一致

### Q: 为什么函数补全会带参数提示或占位符？

这是当前 LSP 辅助设置的一部分。你可以在 **设置 → 语言服务器** 中调整函数参数占位符模式、签名帮助和 Inlay Hints。

### Q: 大项目补全很慢怎么办？

建议顺序：

1. 先确认工程配置没问题
2. 再调低补全候选数量
3. 最后考虑切到远程 LSP
