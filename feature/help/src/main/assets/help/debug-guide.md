# 调试指南

MobileIDE 当前的调试链路以 **LLDB + PRoot** 为核心，重点覆盖“断点、继续、单步、调用栈、变量、调试输出”这些真正高频的调试需求。

## 开始之前要知道什么

调试是否可用，首先取决于：

- 开发环境是否已部署完成
- 当前设备上的 LLDB 是否可用
- 项目本身是否已经能成功构建

如果这些前置条件不成立，调试按钮即使存在，也很难得到稳定结果。

## 当前调试能力的核心范围

基于源码，当前调试会话服务负责：

- 启动 / 停止 LLDB 会话
- 继续、暂停、单步跳过、单步进入、单步跳出
- 暂停时刷新 **调用栈** 和 **变量**
- 把 LLDB 输出透传到 UI，作为调试控制台内容

## 断点

当前工作台支持断点能力，并且编辑器侧已经接入断点相关显示与切换逻辑。

你可以把断点理解为调试流程的入口：

- 在可疑代码位置停住
- 观察当前栈帧
- 查看变量值
- 再决定继续还是单步

## 调试时你会看到什么

### 调试控制

当前 UI 已经明确提供这些操作：

- **继续 / 暂停**
- **单步跳过**
- **单步进入**
- **单步跳出**
- **停止**

### 调试内容页

当前调试内容至少围绕这些信息组织：

- **断点列表**
- **变量**
- **调用栈**

这也是日常排查问题时最有价值的三块信息。

### 调试输出

LLDB 输出会被透传到界面，因此你可以把它当成“调试控制台”来理解，用来观察调试器反馈和程序执行信息。

## 推荐调试流程

一个更贴近当前 MobileIDE 的调试流程通常是：

1. 先确保项目可以正常构建
2. 在可疑位置打断点
3. 启动调试会话
4. 程序停住后查看变量与调用栈
5. 结合继续 / 单步操作缩小问题范围

## 适合用调试器解决的问题

当前最适合用调试器排查的典型问题包括：

- 崩溃位置不明确
- 某个分支为什么没走进去
- 某个变量为什么值不对
- 函数调用顺序和预期不一致
- 多层函数调用里到底是哪里开始出错

## 什么时候先别急着调试

如果当前问题属于下面这些情况，建议先解决基础问题再开调试：

- 项目还没编译成功
- 终端 / PRoot 环境本身就异常
- LLDB 尚未准备好
- 外部依赖路径都还没配对

否则你很容易把“环境问题”误判成“业务逻辑问题”。

## 常见问题

### Q: 调试器起不来？

优先检查：

1. 开发环境是否完整部署
2. LLDB 是否可用
3. 当前项目是否能正常构建成功

### Q: 断点命中了，接下来该看什么？

建议按顺序看：

1. 当前变量值
2. 调用栈
3. 下一步执行路径

### Q: 变量看不懂怎么办？

先不要一次看太多。优先盯住：

- 当前函数输入参数
- 关键中间变量
- 最终控制流程的布尔值或状态值

### Q: 是不是所有问题都该用调试器？

不是。很多构建、路径、工具链或权限问题，其实先看 **构建日志** 和 **终端输出** 更高效。
