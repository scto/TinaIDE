# LSP 概述

LSP（Language Server Protocol）是 MobileIDE 当前代码智能能力的核心基础。对 C / C++ 开发来说，它决定了补全、诊断、跳转和很多“像 IDE 一样”的体验。

## LSP 在 MobileIDE 里负责什么

基于当前实现，语言服务器相关能力主要包括：

- **代码补全**
- **诊断信息**
- **跳转到定义**
- **查找引用**
- **重命名**
- **签名帮助**
- **Inlay Hints**
- **Semantic Tokens（语义高亮）**
- **折叠范围**

其中多数能力都围绕 **clangd** 展开。

## 本地 LSP 与远程 LSP

### 本地 LSP

分析在当前设备完成。

优点：

- 不依赖网络
- 配置相对简单
- 隐私边界更清晰

局限：

- 更吃手机性能
- 大项目索引和补全可能更慢

### 远程 LSP

分析在电脑侧完成，手机主要承担编辑与展示。

优点：

- 更适合大型项目
- 补全和索引通常更快
- 能减轻手机端内存压力

局限：

- 需要网络环境
- 需要额外配置主机、端口和同步方式

## 为什么 `compile_commands.json` 很关键

clangd 想正确理解工程，必须知道真实编译参数。

这份编译数据库会影响：

- 补全准确度
- 头文件解析
- 宏定义识别
- 跳转和引用结果
- 诊断质量

所以当你觉得 LSP “不准”时，第一优先级几乎总是先检查工程构建配置，而不是盲目重启编辑器。

## 设置入口

当前相关入口统一在：

**设置 → 语言服务器**

这里可以看到并调整的内容包括：

- clangd 补全样式
- 函数参数占位符模式
- 补全候选数量
- 补全大小写敏感
- 签名帮助
- Inlay Hints
- Semantic Tokens
- 远程 LSP 主机 / 端口 / 同步模式 / 连接测试

## 推荐理解方式

你可以把 MobileIDE 里的 LSP 理解为：

```text
编辑器输入
   ↓
语言服务器分析工程
   ↓
返回补全 / 诊断 / 跳转 / 签名帮助等结果
   ↓
编辑器负责展示与交互
```

也就是说，编辑器和语言智能是协作关系，而不是全都写死在编辑器里。

## 常见误区

### “我能编译，所以 LSP 一定没问题”

不一定。构建成功只能说明当前构建链路可用，不代表编译数据库、索引状态和语言服务配置都正确。

### “补全慢一定是手机太差”

也不一定。很多时候是：

- 编译数据库不完整
- 项目过大
- 远程 LSP 没配置好
- 当前同步模式不合适

### “LSP 只是补全”

不对。当前 MobileIDE 里的 LSP 还同时承担诊断、跳转、引用、重命名、签名帮助等多项能力。
