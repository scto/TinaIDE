# 远程 LSP 指南

远程 LSP 的目标很明确：**把语言分析压力从手机转移到电脑**，从而让大项目在 MobileIDE 里更好编辑。

## 适合什么场景

更推荐你考虑远程 LSP 的情况：

- 项目很大
- 头文件很多
- 本地补全延迟明显
- 手机内存压力较大
- 你本来就会在电脑侧保留一份工程环境

## 手机端当前可配项

基于当前设置页实现，远程 LSP 相关入口在：

**设置 → 语言服务器**

你可以配置：

- 是否启用远程 LSP
- 主机地址
- 端口
- 同步模式
- 连接测试

应用还会显示连接状态，并根据当前条件检测同步模式。

## 典型配置流程

### 1. 在电脑侧准备语言服务代理

手机端当前会通过网络连接到远程语言服务，因此电脑侧需要提前准备一个可访问的 LSP / WebSocket 代理服务，并确保它能调用 clangd。

### 2. 在手机端填写主机与端口

在 **设置 → 语言服务器** 中：

1. 启用远程 LSP
2. 填写电脑的 IP 或主机名
3. 填写端口
4. 点击 **测试连接**

只有在测试成功后，后续补全和诊断体验才更可靠。

### 3. 选择同步模式

当前实现支持同步模式概念，你需要根据项目规模选择：

- **自动** - 让应用按当前环境判断
- **轻量模式** - 更适合大型工程，减少不必要同步
- **项目模式** - 更适合小型工程或需要完整工程上下文的场景

如果你的工程很大，优先从 **轻量模式** 或 **自动** 开始尝试。

## 什么时候该考虑 rsync

当工程规模持续变大，或者你频繁增量修改很多文件时，可以考虑配合 `rsync` 做更高效的同步。

这通常适合：

- 同一局域网内开发
- 电脑侧已经准备好稳定工作目录
- 你希望减少重复传输开销

## 连接失败时先查什么

### 1. 主机和端口

- IP 是否填写正确
- 端口是否与电脑侧服务一致
- 电脑和手机是否在同一网络环境下

### 2. 电脑侧服务

- 代理服务是否已经启动
- clangd 是否可用
- 防火墙是否拦截对应端口

### 3. 工程上下文

- 远端是否能拿到正确的工程目录
- `compile_commands.json` 是否存在且是最新的
- 当前同步模式是否适合该项目

## 实用建议

### 小项目

不一定需要远程 LSP，本地就够用。

### 中大型项目

优先考虑远程 LSP，收益通常最明显。

### 问题排查顺序

建议按这个顺序排查：

1. 测试连接是否成功
2. 检查远端 clangd 是否可用
3. 检查同步模式是否合适
4. 检查编译数据库是否正确
